# SITP German AI Agent 开发维护与扩展导引

本项目作为同济大学 SITP 科研项目，旨在具备良好的可维护性与可扩展性。本档为后续开发者提供规范建议。

---

## 1. 命名与代码规范 (Coding Standards)

### 1.1 前端 (Frontend)
- **组件命名**: 使用 PascalCase (大驼峰)，如 `StudentDetail.jsx`。
- **状态管理**: 优先使用本地 `useState`。复杂全局状态建议配合 `Context API`（如已实现的 `ToastContext`）。
- **样式**: 统一使用 Tailwind CSS 进行响应式布局。避免编写冗余的 JS 行内样式。
- **API 请求**: 必须通过 `src/api/request.js` 实例发起，禁止直接使用原生 `fetch` 或未配置的 `axios`，以确保 Token 注入。

### 1.2 后端 (Backend)
- **路由函数**: 使用 `snake_case`，动词开头，如 `def submit_exam_answers(...)`。
- **依赖注入**: 统一使用 FastAPI 的 `Depends(get_db)` 获取数据库会话。
- **CRUD 封装**: 所有数据库操作必须在 `crud/repositories.py` 中定义对应的 `CRUD` 类方法，严禁在路由函数内直接编写复杂的 SQL/SQLAlchemy 查询逻辑。

---

## 2. 如何扩展新功能 (How to Extend)

### 2.1 增加一个新的数据表
1. **Model**: 在 `backend/models/entities.py` 中定义 SQLAlchemy 类。
2. **Schema**: 在 `backend/schemas/entities.py` 中定义 Pydantic 验证模型。
3. **Repository**: 在 `backend/crud/repositories.py` 中创建对应的 CRUD 处理类。
4. **API**: 在 `backend/main.py` 中挂载新的路由。

### 2.2 增加一个新的 AI 对话场景
1. **数据库**: 在 `chat_scenes` 表中插入场景元数据（或在 `002_seed_demo.sql` 中添加）。
2. **逻辑**: 在 `POST /api/student/scene-chat` 路由中，根据场景名称（`scene_name`）分支调整 System Prompt。
3. **前端**: 在学生端“学习”模块或“情景任务”列表中增加跳转入口。

## 3. 常见维护任务 (Daily Maintenance)

### 3.1 监控 API 消耗
- 建议定期登录 [Google Cloud Console](https://console.cloud.google.com/) 查看 API 请求成功率与配额消耗情况。

### 3.2 清理过期会话
- `chat_messages` 表随时间推移会快速膨胀。建议定期执行 SQL 清理 30 天前的历史消息：
  ```sql
  DELETE FROM chat_messages WHERE created_at < NOW() - INTERVAL '30 days';
  ```

## 4. 后续演进建议 (Future Roadmap)

- **容器化日志收集**: 引入 ELK 或简单的日志挂载方案，方便追溯 AI 生成失败的具体原因。
- **WebSocket 支持**: 将当前的请求-响应式对话升级为 WebSocket 长连接，实现更平滑的流式打字效果。
- **权限精细化**: 引入真正的 JWT 签名校验（当前为 Mock Token 机制）。
