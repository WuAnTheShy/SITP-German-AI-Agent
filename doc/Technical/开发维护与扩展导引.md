# SITP German AI Agent 开发维护与扩展导引

本文面向后续开发者，目标是让功能迭代时保持结构一致、行为稳定、文档同步。

---

## 1. 代码组织约定

### 1.1 后端分层职责

- routers：仅做参数接收、权限校验、调用服务/仓储，避免写重 SQL 逻辑。
- crud/repositories：聚合数据库读写逻辑，路由层不直接拼复杂查询。
- services：封装跨路由复用能力（如 LLM、指标刷新）。
- core：认证、密码、响应格式等基础能力。
- db/models/schemas：分别负责会话、ORM 模型、请求响应模型。

### 1.2 前端分层职责

- pages：页面级功能编排。
- components：通用组件。
- api：统一请求入口与拦截器。
- context：全局状态（主题等）。
- utils：通用工具函数。

---

## 2. 开发规范

### 2.1 后端

- 函数与变量使用 snake_case。
- 路由统一使用 Depends(get_db) 获取数据库会话。
- 权限校验通过 require_admin、require_teacher、require_student 等统一依赖处理。
- 统一返回结构优先使用 core.responses 中的 ok/fail。

### 2.2 前端

- 组件名使用 PascalCase。
- 所有 API 请求通过 src/api/request.js 发起，避免页面中直接写散乱请求。
- 业务常量优先集中管理，避免魔法字符串散落。

---

## 3. 新功能扩展示例流程

### 3.1 新增一个后端业务接口

1. 在 schemas/entities.py 增加请求与响应模型。
2. 在 models/entities.py 与 db/sql 设计数据结构（必要时补 SQL 脚本）。
3. 在 crud/repositories.py 新增仓储方法。
4. 在对应 routers 文件实现路由逻辑与权限校验。
5. 在 README 与 API 参考手册补充端点说明。

### 3.2 新增一个学生学习模块

1. 后端：在 student_learning.py 或独立 router 增加接口。
2. 前端：在 pages/student 下新增页面并在路由中挂载。
3. 数据：补齐分类表/素材表 seed 或后台管理入口。
4. 指标：必要时调用 metrics 刷新学习行为统计。

### 3.3 新增一个 AI 场景对话

1. 在场景表中写入场景配置。
2. 在 /api/student/scene-chat 的提示词组装逻辑中增加场景分支。
3. 前端在任务中心或学习入口增加跳转。
4. 完成端到端联调：消息发送、上下文保持、清空会话、结果展示。

## 4. 数据与迁移策略

当前项目包含 SQL 初始化脚本与启动时补丁逻辑。迭代数据库时建议：

1. 优先新增版本化 SQL 脚本（db/sql）。
2. 对线上库兼容场景，可在启动阶段做幂等检查后再补丁。
3. 禁止直接修改历史脚本语义，避免旧环境不可重放。
4. 涉及字段重命名或类型变更时，准备回滚 SQL。

---

## 5. 账号与安全约定

1. 管理员、教师、学生密码来源于 .env 或后台重置，禁止代码硬编码默认密码。
2. 登录口令传输与存储必须遵守现有 password 模块约定。
3. 生产环境必须关闭 ALLOW_LEGACY_TOKENS。
4. AUTH_TOKEN_SECRET、QWEN_API_KEY 不可提交到仓库。

---

## 6. 日常维护建议

### 6.1 日志与告警

- 每天巡检 backend 日志是否有持续数据库重连、LLM 调用失败。
- 关注 frontend 日志中的 502，通常对应 backend 不可达。

### 6.2 数据清理

- 会话消息与学习轨迹会持续增长，建议按周期做归档或清理。
- 清理前先做数据库备份，避免误删影响教学追踪。

### 6.3 接口契约检查

- 每次发布后抽样验证关键接口：登录、任务列表、对话、试卷提交、成绩查询。

---

## 7. 提交与发布检查清单

发布前建议至少完成以下检查：

1. 关键流程手工回归：管理员、教师、学生三端登录与核心动作。
2. Docker 启动可通过：docker compose up -d --build 后无致命报错。
3. 数据库兼容性确认：升级部署不会因密码或字段缺失导致启动失败。
4. 文档同步：README、API 参考、运维指南同步更新。
5. 安全项确认：不提交密钥，不暴露调试配置。

---

## 8. 后续演进建议

1. 建立自动化测试基线（至少覆盖登录、鉴权、关键业务路由）。
2. 引入结构化日志与请求追踪，降低线上问题定位成本。
3. 完善数据库迁移工具链，减少启动期动态补丁复杂度。
4. 对高频 AI 路由增加熔断与重试策略，提升可用性。
